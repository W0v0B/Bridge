# v0.2.1

## Bug Fixes

### 1. Drag-drop upload causes path reset and duplicate progress bars

**Problem:** When drag-dropping a file to the File Manager, three identical progress bars appeared (instead of one), and after upload completion the directory view reset to the default path as if the component had reloaded.

**Cause:** The drag-drop `useEffect` included `currentPath`, `loadFiles`, and `uploadModalOpen` in its dependency array. Since `loadFiles` is a `useCallback` depending on `currentPath`, any path navigation caused the effect to re-register Tauri event listeners. Due to the async nature of `listen().then()`, old listeners could leak â€” the cleanup function runs before the `.then()` resolves, so old listeners were never unsubscribed. Each leaked listener independently called `pushFiles` on the same drop event, producing duplicate transfers and progress bars. Stale `loadFiles` references in leaked closures also captured old `currentPath` values, causing the apparent path reset.

**Fix:** Replaced closure-captured values with `useRef` for `currentPath`, `uploadModalOpen`, `loadFiles`, and `selectedDevice`/`connectKey`. The effect now only depends on the device identifier, so listeners are registered once per device. An `active` flag prevents leaked listeners from firing, and a `cleanup` helper immediately unsubscribes any listener that resolves after the effect has been torn down.

### 2. Upload modal drag triggers File Manager overlay

**Problem:** When dragging files onto the Upload Modal's drop zone, the File Manager's dashed-border drag overlay also appeared behind the modal.

**Cause:** Tauri drag events (`tauri://drag-enter/leave/drop`) are window-level â€” both the UploadModal and FileManager listeners receive every drag event. The FileManager's `if (uploadModalOpen) return;` guard relied on a closure-captured value that could be stale due to the listener leak described above.

**Fix:** The FileManager now reads `uploadModalOpenRef.current` (a ref synced every render) instead of a closure-captured state value, ensuring the guard always reflects the current modal state regardless of when the listener was created.

### 3. Cross-device upload independence verified

**Check:** Uploads continue in the background when switching devices. The backend `push_files`/`send_hdc_files` commands run independently per device serial. After upload completion, `loadFiles()` is only called if the user is still viewing the same device that initiated the upload, preventing stale refreshes.
