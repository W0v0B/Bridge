# v0.3.2

## Performance

### Shell Terminal — Incremental ANSI Rendering

Eliminated render lag that appeared after ~3000 lines of output in the Shell terminal.

**Root cause**: each incoming chunk triggered `ansiToHtml(entireOutput)` — an O(n) full-string parse — followed by a React state update that re-rendered the entire output `<div>`.

**Fix**:
- Per-device `AnsiConverter` instances parse ANSI escape sequences incrementally, preserving SGR color state (fg/bg/bold/italic/underline/dim) across chunks. Only the new chunk is parsed on each write — O(chunk size).
- Per-device HTML is stored in `htmlStringMap`/`htmlChunksMap` refs and written directly to the output `<div>` via `innerHTML`, bypassing React state for output rendering entirely.
- Non-selected devices accumulate data silently without touching the DOM.

### Logcat / HiLog — Streaming DOM Append

Eliminated render lag that appeared after ~3000 log entries in Logcat and HiLog.

**Root cause**: each RAF frame replaced the entire `innerHTML` of the log content `<div>` — O(all entries) — and a custom string-scanning function (`trimHtml`) trimmed the HTML buffer O(n) on every write.

**Fix**:
- **Streaming path** (`flushToDOM`): new entries are appended via `insertAdjacentHTML("beforeend", …)` — O(new entries) per frame. Excess children are removed directly from the DOM, no string scanning.
- **Rebuild path** (`rebuildAndFlush`): replaces `innerHTML` from the bounded in-memory entry array — O(max lines). Used only on filter/device/mode changes and when catching up after auto-scroll resumes.
- Pending HTML buffer is discarded immediately when auto-scroll is paused (data is retained in `buf.entries`); `rebuildAndFlush` handles catch-up from the bounded entry array.

## Bug Fixes

### Logcat / HiLog — Scroll Hang on Resume

**Problem**: scrolling back to the bottom while paused called `flushToDOM` synchronously from the scroll handler, causing the browser to parse thousands of accumulated DOM nodes at once and then remove excess children one-by-one in a loop — resulting in a multi-second freeze.

**Fix**: `handleScroll` now defers `rebuildAndFlush` to the next animation frame via `cancelAnimationFrame` + `requestAnimationFrame`. This prevents synchronous heavy DOM work inside a scroll handler, which could also be triggered by scroll-anchoring events mid-stream.

### Shell Terminal — "Stopping..." Feedback

**Problem**: clicking Stop on a high-throughput command (e.g. `logcat`) showed no feedback — the Stop button remained active and the input stayed disabled for 4–5 seconds while the OS pipe buffer drained.

**Fix**: the Stop button transitions to a "Stopping..." state (spinner icon, dimmed, non-interactive) immediately on click. Input remains disabled until `shell_exit` confirms process death, preventing a backend race condition that would occur if a new command were started mid-drain. Buffered output continues to display during drain.

### Shell Terminal — Stop Button Contrast in Snow Theme

**Problem**: in the Snow (light) theme, the terminal input bar always has a dark background (`var(--term-header-bg)`). Ant Design's `disabled` prop desaturates a `danger type="primary"` button to gray under the light algorithm, which clashed visually with the dark background.

**Fix**: the Stopping state no longer uses `disabled`. Instead, `pointerEvents: "none"` + `opacity: 0.65` are applied via inline style, keeping the button red and readable on the dark terminal background across all themes.

### App Manager — Theme Switch Style Warning

**Problem**: switching to the Snow theme in the Apps section triggered a React warning: *"Removing a style property during rerender (background) when a conflicting property is set (backgroundColor)"*. Dark-theme `Tag` elements used the `background` shorthand in their `style` prop; light-theme `Tag` elements used Ant Design's `color` prop, which internally sets `backgroundColor`. React detected the shorthand/non-shorthand conflict on the same DOM element.

**Fix**: replaced `background` with `backgroundColor` in all dark-mode `Tag` inline styles in `AppManager.tsx`.
