# v0.2.2

## Bug Fixes

### 1. Upload success message shown on failure

**Problem:** When a file upload failed (e.g., permission denied), the progress bar still showed 100% success (green). The backend emitted a 100% completion event before checking if the command actually succeeded.

**Cause:** In both `adb/file.rs` and `hdc/file.rs`, the `transfer_progress` event with `percent: 100.0` was emitted unconditionally before the `status.success()` check. This caused the TransferQueue progress bar to show green 100% even on failure.

**Fix:** Moved the 100% emission to after the success check. On failure, a progress event with `percent: -1.0` and `speed: "failed"` is emitted instead. The TransferQueue frontend now handles this failure state, showing a red error bar and auto-removing it after 3 seconds.

### 2. OHOS multi-device remount not working

**Problem:** When multiple OHOS devices were connected, the automatic remount function did not work reliably for all devices.

**Cause:** The remount only ran `hdc -t <key> target mount`, but a successful remount requires both `shell mount -o rw,remount /` and `target mount` to be executed.

**Fix:** Changed `attempt_remount` to run both commands sequentially: first `hdc -t <key> shell mount -o rw,remount /`, then `hdc -t <key> target mount`. Both must succeed for the remount to be considered successful. Also added `Read-only file system` to the failure marker detection.

### 3. HdcCatModal white background (theme mismatch)

**Problem:** The file viewer modal on OHOS devices had hardcoded light-theme colors (`#f5f5f5` background, `#d9d9d9` border, white loading overlay), making it look broken in dark theme.

**Cause:** The HdcCatModal used hardcoded light-theme hex colors instead of CSS variables, unlike the ADB CatModal which correctly used `var(--card-bg)`, `var(--border)`, and `var(--text-primary)`.

**Fix:** Replaced all hardcoded colors with CSS variable equivalents matching the ADB CatModal implementation.

### 4. OHOS device disconnect not working

**Problem:** Clicking disconnect on an OHOS device just removed it from the UI, but the device watcher re-discovered it and re-added it immediately.

**Cause:** No backend disconnect function existed for OHOS devices. The Sidebar's `handleDisconnect` simply called `removeDevice()` (UI-only removal), while the device watcher continuously polls `hdc list targets` and re-emits any still-connected device.

**Fix:** Added `disconnect_device()` function in `hdc/device.rs` using `hdc tconn <addr> -remove` (the documented HDC disconnect command). Registered it as a Tauri command (`disconnect_ohos_device`) and wired the Sidebar to call it for OHOS devices. The function also cleans up cached remount status for the disconnected device.
