# v0.3.6

## New Features

### Script Execution — Interactive Input & Enter Button

A **↵ Enter** button is now permanently visible in the Shell toolbar (no longer hidden behind a running state):
- **Serial devices**: sends `\r\n` via `writeToPort` — useful for confirming prompts at any time
- **ADB / OHOS devices**: injects `\n` into the running script's stdin via `sendScriptInput` — allows responding to interactive prompts (e.g. `pause`, `read`, `Confirm?`) without stopping the script

### Script Execution — `.sh` Line-by-Line Dispatch

`.sh` scripts in Quick Commands are now sent line-by-line to the connected device instead of running locally:
- **ADB**: each non-blank line is dispatched via `startShellStream(serial, line)` with a 200 ms gap between lines
- **OHOS**: each line dispatched via `startHdcShellStream(connect_key, line)`
- **Serial**: each line written via `writeToPort(serial, line + "\r\n")`

This allows shell scripts to be authored on the PC and executed directly on the target device without manually pushing them first.

### Script Execution — PowerShell UTF-8 Encoding

`.ps1` scripts now run via `powershell -ExecutionPolicy Bypass -NoProfile -Command "[Console]::OutputEncoding=UTF8; [Console]::InputEncoding=UTF8; & '<path>'"` instead of `-File`. This ensures non-ASCII output (Chinese characters, etc.) is correctly encoded as UTF-8 before the backend receives it.

## Bug Fixes

### Chinese Characters Garbled in Shell Terminal

**Problem**: On Chinese Windows systems, `cmd.exe` and other processes emit output in the system OEM codepage (e.g. GBK, code page 936) rather than UTF-8. The backend was decoding all subprocess bytes with `String::from_utf8_lossy()`, which replaces invalid UTF-8 sequences with U+FFFD, corrupting any non-ASCII characters.

**Fix**: All subprocess byte-to-string conversions now go through `decode_process_output()`:
1. Try UTF-8 first (fast path — covers ADB/HDC device output, which is always UTF-8)
2. On failure, call `GetOEMCP()` (Windows API) to detect the system codepage, map it to an `encoding_rs` encoding (936/54936 → GBK, 950 → Big5, 949 → EUC-KR, 932 → Shift_JIS), and decode the bytes correctly

Applied across all output paths: `adb/commands.rs`, `hdc/commands.rs`, `adb/device.rs`, `adb/apps.rs`, `hdc/device.rs`, `hdc/apps.rs`, `adb/logcat.rs`, `hdc/hilog.rs`, `util.rs`.

### Stop Button — Script Not Killed When Shell Stream Absent

**Problem**: The Stop button called `stopShellStream` and `stopLocalScript` in the same `try/catch`. If `stopShellStream` threw (no active stream), `stopLocalScript` was never called, leaving the script running.

**Fix**: Each call now has its own independent `.catch(() => {})` so a failure in one never blocks the other.

### Stopping Indicator Stuck After Script Exit

**Problem**: The `script_exit` event handler updated `setDeviceRunning` but did not reset `stoppingMap` or call `setStopping(false)`, leaving the toolbar stuck at "Stopping…" after the script exited naturally.

**Fix**: `script_exit` now resets both `stoppingMap.current[id]` and `setStopping(false)`, matching the behaviour of the `shell_exit` and `hdc_shell_exit` handlers.

### Zombie Script Processes on Device Disconnect

**Problem**: If a serial device was disconnected (or a device was manually disconnected) while a `.bat`/`.ps1` script was running, the script process continued in the background holding the serial port open. The port would remain occupied until the app restarted.

**Fix**: `stopLocalScript(device.id)` is now called (fire-and-forget) before any device disconnect:
- `Sidebar.handleDisconnect()` — covers all user-initiated disconnects (ADB, OHOS, serial)
- `useSerialDisconnect()` — covers unexpected serial disconnects via the `serial_disconnected` event
