# Version 0.1.1

## Bug Fixes

### 1. Logcat/Hilog panel not independent per device

**Problem:** When multiple devices were connected, the Logcat panel shared a single log buffer across all devices. Switching devices would display logs from the previously viewed device. The Hilog panel on OHOS devices had the same issue.

**Cause:** The backend emitted `logcat_lines`, `tlogcat_lines`, and `hilog_lines` events without including the device serial/connect_key, making the frontend unable to distinguish which device the logs came from. The frontend maintained a single set of buffers (per-mode only, not per-device).

**Fix:**
- **Backend:** Added `LogcatBatch` (with `serial` field) and `HilogBatch` (with `connect_key` field) wrapper structs. All log event emissions now include the device identifier.
- **Frontend:** Changed event hooks to receive the new batch format. Rewrote `LogcatPanel` and `HilogPanel` to maintain per-device buffers keyed by `"serial:mode"` (for logcat/tlogcat) or `"connect_key"` (for hilog). Running state is also per-device. Switching devices now shows the correct independent log buffer for each device.

### 2. tlogcat provides no feedback when unsupported

**Problem:** If a device did not support `tlogcat` (no TEE environment), the panel showed no output or error. The user had no way to know why nothing appeared.

**Cause:** The tlogcat process was spawned with `stderr(Stdio::null())`, discarding all error output. When the shell command failed (e.g., `sh: tlogcat: inaccessible or not found`), the error message was silently dropped.

**Fix:** Changed `stderr` from `Stdio::null()` to `Stdio::piped()` in `start_tlogcat`. Added a dedicated tokio task that reads stderr lines and emits them as error-level `tlogcat_lines` entries (level "E", tag "tlogcat"). Error messages now appear directly in the tlogcat panel.

### 3. File Manager shares current path across devices

**Problem:** When switching between devices, the File Manager retained the current directory from the previous device. For example, navigating to `/data` on device 1 then switching to device 2 would directly open `/data` on device 2.

**Cause:** The `currentPath` state was a simple `useState` with no per-device storage.

**Fix:** Added a per-device path map (`useRef<Record<string, string>>`) to both `FileManager` (ADB) and `HdcFileManager` (OHOS). On device switch, the current path is saved to the map for the old device and restored from the map for the new device (defaulting to `/sdcard` for ADB, `/data` for OHOS).

### 4. Shell terminal does not support ANSI color codes

**Problem:** When commands output ANSI-colored text (e.g., `\033[32mPASS\033[0m`), the raw escape sequences were displayed as plain text instead of rendering as colored output.

**Cause:** Shell output was rendered as plain text using React text content, with no ANSI escape code parsing.

**Fix:**
- Created `src/utils/ansi.ts` — a lightweight ANSI-to-HTML converter supporting SGR sequences (16 colors, 256 colors, true color, bold, dim, italic, underline).
- Added a toggle button (color palette icon) in the Shell panel header to enable/disable ANSI color rendering. Enabled by default, per-device.
- When enabled, output is rendered via `dangerouslySetInnerHTML` with HTML-escaped and ANSI-converted content.
- When disabled, ANSI codes are stripped and output is shown as plain text.
